# Slay the Spire 2 AI Agent Project (STS2LLM)

> This file is the living plan for this project. Update it whenever the plan changes or a phase completes — do not let it go stale. Check it before starting new work in a session.

## Context

The user's existing `E:\BalatroLLM` project (Lua mod + Python MCP stdio server + strategy-pack docs) lets Claude autonomously play Balatro via MCP tool calls. The user now wants the same capability for **Slay the Spire 2**: an MCP tool line so an LLM can climb the tower by itself, plus (secondarily) an RL-trained policy — both ultimately aimed at "let AI play this game."

Research turned up a mature, actively-maintained open-source ecosystem already solving most of this:

- **[STS2MCP](https://github.com/Gennadiyev/STS2MCP)** (432★, 76 forks, **MIT license**, last pushed 2026-06-16) — a C# mod that starts an HTTP server (`localhost:15526`) exposing full game state + actions, plus a Python **FastMCP** server (`mcp/server.py`) bridging it to any MCP client. It already implements **65 tools** covering menu, combat, map, shop, events, rewards, relics, potions, deck/bundle selection, the crystal-sphere minigame, and full multiplayer — singleplayer is fully supported and tested against a recent STS2 build. It even ships `AGENTS.md` (a strategy/conventions guide for the LLM, e.g. index-shift gotchas when playing/claiming cards right-to-left) and a `.claude/commands/playsts2.md` slash command for driving an interactive Claude Code session. This is architecturally the same shape as BalatroLLM (mod + MCP bridge) and is **already a working, better-tested equivalent** — rebuilding it from scratch would just re-solve problems (game-hook maintenance across STS2 patches, animation/state-timing, index-shift bugs) this project has already worked through.
- **[sts2-rl-agent](https://github.com/zhiyue/sts2-rl-agent)** (33★, last pushed 2026-05-22) — headless Python simulator (100% card/relic/monster/power coverage, ~1200 combats/sec) + Gymnasium env + MaskablePPO (Stable-Baselines3) training + a separate Harmony-patched C# bridge mod to run the trained policy in the real game. **No LICENSE file** — confirmed 404 on `/LICENSE`. This means it can be studied as an architecture reference but its code must **not** be vendored/copied into our repo.
- TokenSpire2 and sts2-modding-mcp were also reviewed but are less relevant (TokenSpire2 bypasses MCP entirely via OpenRouter; sts2-modding-mcp is a dev-tooling MCP for *building* mods, not playing).

Given this, the plan is to build on top of STS2MCP rather than duplicate it, and treat the RL line as a clean-room build informed by (but not copying) sts2-rl-agent's public design.

## Recommended Approach

New repo at **`E:\STS2LLM`** (separate from BalatroLLM, per user's preference).

### Phase A — MCP tool line (priority)

1. **Vendor STS2MCP as the base.** Clone `Gennadiyev/STS2MCP` (MIT) into `E:\STS2LLM`, keeping its LICENSE/attribution intact. Build the C# mod (`STS2_MCP.csproj`, needs .NET 9 SDK + game install as build reference) and deploy the DLL + manifest to the STS2 `mods/` folder, mirroring how `tools/deploy_mod.ps1` deploys the Balatro mod today.
2. **Verify end-to-end.** Confirm `curl http://localhost:15526/health`, then configure Claude Code to use `mcp/server.py` (same pattern as `HOW_TO_CONNECT.md` for Balatro), and run one full interactive session using the existing `.claude/commands/playsts2.md` — menu → map → combat → shop/rewards → at least one boss — to confirm the tool surface works against the user's current STS2 version.
3. **Port over BalatroLLM patterns that add value**, not code (STS2MCP is C#/Godot, Balatro mod is Lua/LÖVE — no direct code reuse):
   - A `strategy/` playbook pack (per-character: Ironclad/Silent/Defect/Watcher/5th class), loaded contextually rather than dumped in the system prompt — same idea as `balatro_mcp/agent/strategy_loader.py`, extending STS2MCP's existing generic `AGENTS.md` advice with deck-specific playbooks.
   - Deploy/setup scripts and a `HOW_TO_CONNECT.md`-style doc adapted for STS2's mod folder layout and FastMCP config.
4. **New capability not in STS2MCP: an unattended batch-run harness.** STS2MCP's own design is for a human-supervised interactive Claude Code session (`.claude/commands/playsts2.md` explicitly uses second-person, human-judgment language — confirmed not an autonomous loop). To get genuinely hands-off "AI climbs the tower on its own" runs (and to produce data/eval logs later reusable by the RL line), add a Python harness under `tools/` that drives the Anthropic API/Claude Agent SDK in a loop directly against the mod's HTTP API: start run → call Claude with `get_game_state` + legal actions → execute chosen action → repeat until game over → log outcome (floor reached, deck, cause of death) → restart, for N runs unattended.

### Phase B — RL line (secondary, after Phase A works)

1. Treat sts2-rl-agent's README-documented architecture (headless sim / Gym env / MaskablePPO / reward shape) as a **reference design only** — no code copied, since it has no license.
2. Build a clean-room headless simulator + Gymnasium environment in `rl/`, bootstrapping card/relic/monster/power data from **STS2MCP's own MIT-licensed runtime endpoints** (`get_compendium`, `search_wiki`) rather than transcribing from an unlicensed source.
3. Train with MaskablePPO (Stable-Baselines3) using a similar reward shape as a starting point (win/loss ±1, optional HP-loss penalty, floor/act progression bonus), iterating from there.
4. Bridge the trained policy into the real game by reusing **Phase A's own STS2MCP HTTP API** (it already exposes full state + actions) instead of building a second bridge mod — simpler than sts2-rl-agent's two-mod design, since one mod already does the job.
