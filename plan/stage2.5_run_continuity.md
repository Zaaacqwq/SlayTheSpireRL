# 我的诚实评估 + Stage 2.5 提案：连续战斗 + 简单卡牌奖励

## 先直接回答用户的问题

用户在看到 Stage 2 训练曲线后追问：训练用的是固定10张起手牌、每场都是孤立满血战斗，这样训练真的有用吗，还是只是当前阶段、后续会改进？

**现在的训练是"有用但很窄"，不是"没用"，也不只是"当前阶段自然会变好"——如果不主动改，光靠 Stage 3/4/5（加职业、加怪物、加遗物药水）也不会解决这个问题。**

具体拆开说：

- **训练循环本身是真的在学东西**：胜率/reward 会随训练上升，这证明了模拟器规则、Gym 包装、PPO 训练链路都是正确工作的。这个价值是实的，不是幻觉。
- **但训练出来的策略学的是"怎么用固定的10张起手牌打一场孤立的满血战斗"，不是"怎么打一局 STS2"**。用户指出的两个问题都对，而且是同一个根源：v1/Stage 1/Stage 2 的范围从一开始就明确排除了"地图/连续战斗/卡牌奖励"（见 `plan/stage0_v1_single_combat.md`），这是当初为了先把"模拟器规则对不对"这个更基础的问题验证清楚而做的刻意收窄，不是疏忽。
- **单靠继续做 Stage 3（加职业）/Stage 4（加怪物）/Stage 5（加遗物药水）不会自动修好这个问题**——那些都是在"content"维度加东西，训练时摸到的还是那10张固定卡，打的还是孤立的一场满血战斗。真正解决这个问题，路线图里对应的是 **Stage 6（完整 run 结构）**，但那是目前列出来"工作量最大的单一里程碑"（地图生成+商店+事件+分层动作空间+稀疏reward），如果直接跳过去做全套，是一次大投入。

所以结论：**现在的训练成果是"确认了地基没问题"，不是"已经在训练一个会打 STS2 的 agent"，而且如果不主动调整优先级，光靠往后做 Stage 3/4/5 并不会让训练结果变得更有代表性。**

## 推荐方案：不是"全套 Stage 6"，也不是"什么都不改"，而是一个折中的 Stage 2.5

直接上完整 Stage 6（地图选路+商店+事件+遗物+分层动作空间）工作量很大，而且地图/商店这些其实不是用户这次提出的核心顾虑——用户说的是两件更具体的事：**牌库不会变化**、**战斗不连续（每场都满血重开）**。这两件事可以用一个小得多的中间方案解决，不需要先把地图/商店/事件系统全建起来。

**Stage 2.5：连续战斗 + 简单卡牌奖励（run 连续性的最小可行版本）**

- 一个 episode 从"一场战斗"变成"一串连续战斗"（固定 5 场，怪物随机挑选）。
- 战斗之间 **HP 不重置**——沿用上一场结束时的血量进入下一场，真实还原"资源会被消耗、需要节约"这个 STS 的核心张力。
- 每场战斗结束后（如果没死），进入一个**简单的选卡奖励步骤**：从当前卡池随机抽 3 张，agent 选 1 张加入牌库或跳过——这是新增的一种动作类型，比真实游戏简化（真实游戏还有金币/遗物/事件三选一之类），但已经足以让"卡组会演化"这件事成立。
- 结束条件：连续打完预定场次算这次 run 成功；HP 归零算失败。
- **不做**：地图选路（直接给定固定的战斗序列，不给选择）、商店、事件、遗物、药水——这些还是留给原来定义的 Stage 6，那时候再把地图/商店这层加上去。

这个范围能直接解决用户提的两个问题（牌库演化、连续消耗），工作量比完整 Stage 6 小很多，但比"现在这样"要真实得多。

### 需要的改动
- **`rl/sim/run.py`（新建）**：`Run` 类管理"当前第几场、玩家HP结转、卡组、下一步是combat还是选卡"，仿照 `sim/combat.py` 的纯 Python 状态机写法，不绕 Gym API，单独可测试。
- **`rl/env.py`**：从"每次 reset() = 一场战斗"改成"每次 reset() = 一串战斗+选卡"的小状态机：env 内部靠 `Run.phase`（"combat"/"reward_pick"）决定当前合法动作；动作空间用固定大小的 discrete 空间覆盖"战斗内动作"和"选卡奖励动作"的并集；观测空间加当前第几场/还剩几场、当前 phase、选卡阶段的 3 张候选卡特征。
- **`rl/sim/combat.py`**：不需要改动——`Combat` 类本来就是"一场战斗"的状态机，`Run` 只是在外面套一层编排逻辑。
- **`rl/train.py`**：reward 曲线的意义会变化，需要注意 outcome 字符串从 `"win"/"loss"` 变成 `"run_won"/"run_lost"`。

## Verification

1. `pytest rl/tests/`：新增 `Run` 类的单元测试（连续两场战斗 HP 结转不重置、选卡奖励确实进入下一场牌库、run_won/run_lost 判定正确、跨阶段动作非法应报错）。
2. `gymnasium.utils.env_checker.check_env()`：改完的分阶段 env 依然符合 Gym API。
3. 随机 fuzz 测试：跨若干随机 mini-run（战斗+选卡都随机选合法动作），零报错，且验证 HP 确实跨战斗结转、被选中的奖励卡确实出现在后续战斗的牌库里。
4. 跑一次训练（100k 步左右，跟之前几次同规格），生成新的训练曲线，加进 Stage 2.5 的记录里。

## 实际实现结果

按计划完成，`rl/sim/run.py` 新建 `Run` 类（240行），`rl/env.py` 重构为双阶段状态机（动作空间 41→45，观测空间 112→126维，新增"当前phase"/"第几场"/"3张候选卡特征"）。

**验证**：
- `Run` 类新增 6 条单测全部一次性通过（HP结转、奖励卡拾取、跳过奖励、战败判定、通关判定、跨阶段非法动作报错）。
- `check_env()` 通过。
- 500 局随机legal-action fuzz测试通过 Gym 接口跑通，**0 报错，胜率 23%**（对比 Stage 1/2 单场战斗下随机策略胜率接近100%，证明连续5场+HP不重置真的把任务变难了，符合设计目标）。
- 额外用确定性种子验证了 HP 结转在真实 env.step() 调用链路下完全生效（fight 0 结束69血，fight 1 开局也是69血，非满血重置）。

**训练结果（100k步，跟之前几次同规格）**：初次训练跑出来"训练前后胜率都是0%"——排查发现是 `train.py` 的 `WinRateCallback`/`evaluate()` 还在检查旧的 `"win"/"loss"` 字符串，而新 env 现在上报的是 `"run_won"/"run_lost"`，导致胜率统计从头到尾都没匹配上（真实训练是有效的，只是被这个bug完全遮住了看不见）。修复字符串后重跑：**训练前通关率 25%，训练后稳定在 98-100%**（第46000步左右开始饱和），**平均 reward 从 3.156 涨到 8.350**，曲线呈现清晰的"学习中"S形（不像 Stage 1/2 那样一开始就顶格）。这次是路线图里第一条"真正在学一个更接近真实游戏挑战"的训练曲线。已加进训练曲线 artifact 的跨 Stage 对比表格。

**明确排除，仍留给 Stage 6**：地图选路（战斗序列是固定随机的，不给玩家/agent 选择权）、商店、事件、遗物、药水、多角色、更高难度。
